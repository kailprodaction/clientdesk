# ClientDesk — AI-портал для фрилансеров и небольших агентств

## Идея в одном абзаце

**ClientDesk** — лёгкий SaaS-портал, где фрилансер или небольшая студия ведёт клиентские проекты в одном месте: делится макетами, файлами и прогрессом, собирает **точечные комментарии клиента прямо на изображении или PDF** (клик по координате → тред обсуждения), автоматически превращает git-коммиты и закрытые задачи в **человеческий статус-отчёт для клиента** через Claude API и выставляет счета. Основной оффер — «клиент видит прогресс без созвонов, ты не тратишь час в неделю на статус-письма».

Целевая аудитория: дизайнеры, разработчики-фрилансеры, маленькие продуктовые студии (2–8 человек), которые ведут 3–15 клиентов одновременно и устают от Slack-хаоса, Google Docs и вечных «а где мы сейчас?».

## Почему такой стек

Продукт по природе — CRUD с файлами, комментариями и парой AI-эндпоинтов. Не нужен ни SPA, ни тяжёлый Django. Оптимальное решение:

- **Flask** — минимум магии, быстрый холодный старт под Cloud Run.
- **Supabase** — Postgres + Auth + Storage + realtime в одном пакете; нет своего сервиса аутентификации и хранения.
- **Cloud Run** — платим за использование, autoscale от 0, простой деплой через Docker.
- **Alpine.js + Tailwind + Jinja2** — интерактив без SPA, весь HTML собирается на сервере, минимум JS в бандле, страницы летают.
- **Claude API** — суммаризация, черновики, категоризация фидбека.

Итого: маленькая команда, маленький счёт за инфру, никакой сложности управления состоянием.

## Фичи (MVP → v1)

**MVP:**
1. Регистрация и рабочее пространство (workspace) на команду.
2. Проекты: клиент, дедлайн, статус, участники.
3. Загрузка файлов (изображения, PDF, видео) в проект.
4. **Точечные комментарии** — клик по координате на изображении/PDF → тред.
5. Список задач в проекте (Kanban: Todo / In Progress / Review / Done).
6. Клиентский view (по magic-link, без регистрации) — только чтение + возможность комментировать.

**v1:**
7. **AI Status Report** — раз в неделю Claude собирает список закрытых задач + git-коммитов (если подключён GitHub) и пишет клиенту статус-письмо человеческим языком.
8. **AI Feedback Summary** — все комментарии клиента группируются в actionable-список задач.
9. Тарификация Stripe (Free / Pro / Studio по числу активных проектов).
10. Транзакционные письма через Resend (комментарий, статус, приглашение).

### Слой данных (Supabase / Postgres)

Ключевые таблицы:

- `workspaces` — рабочее пространство (агентство/фрилансер).
- `workspace_members` — участники, роли (`owner`, `member`).
- `projects` — проект (workspace_id, client_name, deadline, status).
- `files` — метаданные файлов в Storage (path, mime, project_id).
- `comments` — position_x, position_y, page (для PDF), file_id, parent_id, body.
- `tasks` — status (`todo/in_progress/review/done`), assignee_id, project_id.
- `client_links` — magic-link токены для клиентского view.
- `status_reports` — сгенерированные отчёты (period, content, sent_at).
- `subscriptions` — Stripe customer_id, план, лимиты.
- `audit_log` — на кто/что/когда для чувствительных действий.

**Multi-tenancy через RLS:** каждая таблица имеет `workspace_id`, политики RLS проверяют, что `auth.uid()` состоит в `workspace_members` для этого workspace. Клиентские magic-links реализованы отдельной политикой на `client_links.token` → доступ к чтению проекта и создание `comments`.

### Backend (Flask)

Структура:

```
app/
├── __init__.py           # create_app, blueprints, config
├── config.py             # 12-factor через env
├── extensions.py         # supabase client, stripe, sentry
├── auth/                 # magic link, session, RLS-aware queries
├── projects/             # CRUD проектов, файлов, задач
├── comments/             # позиционные комментарии + realtime
├── ai/                   # обёртки над Claude (summaries, reports)
├── billing/              # Stripe checkout, webhook
├── webhooks/             # GitHub, Supabase auth events
├── templates/            # Jinja2
├── static/
└── tasks/                # обработчики Cloud Tasks
```

Ключевые решения:
- Один `SupabaseClient` в контексте запроса, инициализируется JWT текущего пользователя — RLS работает автоматически.
- Все внешние вызовы (Claude, Stripe, Resend) в отдельных сервис-классах, легко мокаются в тестах.
- Idempotency-key для Stripe и Cloud Tasks webhook.

### Фронтенд

- **Jinja2 + Tailwind** — компоненты через `include`/`macro`, дизайн-система в `tailwind.config.js` (палитра, тени, радиусы, шрифты).
- **Alpine.js** — там, где нужен интерактив: dropdown, модалки, drag-and-drop задач в Kanban, тред-панель комментариев, upload с прогрессом.
- **Позиционные комментарии:** `<canvas>` поверх изображения/PDF (PDF.js), клик → координаты в процентах от размера файла (не в px, чтобы корректно работало на любом экране), POST в API, оптимистичное обновление.
- **Realtime:** Supabase Realtime channel на `comments` для конкретного файла — новые комментарии появляются без перезагрузки.

### Фоновые задачи

Cloud Scheduler → HTTPS endpoint на Cloud Run → задача попадает в очередь Cloud Tasks → воркер-эндпоинт обрабатывает:

- Еженедельная генерация `status_reports` для активных проектов.
- Отложенные напоминания клиенту о дедлайне.
- Периодическая синхронизация GitHub коммитов.
- Дневная агрегация usage для биллинга.

### AI-слой (Claude)

Два основных сценария:

1. **Status report.** На вход: список закрытых задач за неделю + git-коммиты (если подключён GitHub) + прошлый отчёт. Промпт: «Напиши статус-письмо клиенту в дружелюбном, но профессиональном тоне, без жаргона, 3–5 абзацев, укажи что сделано, что в работе, что ожидает клиента». Ответ сохраняется как draft → владелец может отредактировать → «Отправить».
2. **Feedback summary.** На вход: все комментарии клиента в проекте с координатами и файлами. Промпт: «Сгруппируй по темам, вытащи actionable-пункты в формате чек-листа с приоритетом». Ответ рендерится в UI как список задач с кнопкой «Создать задачу».

Все AI-вызовы асинхронные (Cloud Tasks), пользователь видит «Готовлю отчёт…», уведомление приходит через realtime channel.

### Безопасность

- Все запросы через HTTPS, HSTS.
- CSRF на всех POST-формах (Flask-WTF).
- RLS на уровне БД — даже баг в приложении не даст утечь данным между workspaces.
- Magic-link токены с TTL и одноразовые для инвайтов.
- Secret Manager для ключей (Stripe, Claude, Resend, Supabase service key).
- Rate limiting на публичных эндпоинтах (Flask-Limiter + Redis / Memorystore).

### Мониторинг и надёжность

- **Sentry** — исключения и трейсинг.
- **Cloud Logging + Log-based alerts** — 5xx, ошибки Stripe webhook.
- **Uptime checks** на `/healthz`.
- **Structured logs** в JSON, каждый запрос помечен `request_id` и `workspace_id`.

### CI/CD

- GitHub Actions: `pytest` + `ruff` + `mypy` на PR.
- На main-ветке: сборка Docker → push в Artifact Registry → `gcloud run deploy` с новым revision.
- Preview-окружения на каждый PR через Cloud Run tags.
- Миграции Supabase применяются отдельным job'ом с проверкой на breaking-changes.

### Тесты

- `pytest` для unit + integration (test-контейнер с Postgres).
- Playwright для e2e ключевых флоу: логин → создание проекта → загрузка файла → комментарий → статус-отчёт.
- Coverage ~80% на бизнес-логику, критические места (биллинг, RLS) — 100%.

## Метрики / что мерил в проекте

- **TTFB:** ~180 мс (Cloud Run, europe-west1).
- **P95 latency** дашборд-страницы: ~350 мс.
- **Холодный старт** Cloud Run: ~1.2 с (min-instances=0, для платных workspace min-instances=1).
- **Инфра-косты MVP:** ~$40/мес на первые 200 workspaces.

## Что бы сделал иначе

Если бы начинал с нуля — сразу подключил бы Supabase Edge Functions для webhook-эндпоинтов (Stripe, GitHub), чтобы не гонять Cloud Run на мелких пингах. И раньше бы вложился в feature flags — гоняли релизы через `main`, пару раз ловили баг на проде из-за отсутствия дарк-лаунча.
